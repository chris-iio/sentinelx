## Summary

This slice adds `IOCType.EMAIL` to the extraction pipeline and displays email IOCs in the results page with no enrichment. The work is straightforward ‚Äî iocsearcher already extracts emails (with `name=email`), the normalizer already defangs `[@]`/`[at]`/`(@)` ‚Üí `@`, and the results template auto-generates type filter pills from `grouped.keys()`. The main gap is that `classifier.py` has no email case, so extracted emails are silently discarded by `run_pipeline()`.

One non-obvious pitfall: **OTX adapter uses `frozenset(IOCType)` for `supported_types`** (line 85 of `otx.py`), meaning it dynamically includes ALL enum values. Adding `EMAIL` to `IOCType` will make OTX claim it supports emails, and its `_OTX_TYPE_MAP` lookup will raise `KeyError` since there's no email‚Üípath mapping. This must be fixed by changing OTX to an explicit type list.

## Recommendation

Add `EMAIL = "email"` to `IOCType`, add an email regex + classify case, fix OTX's `supported_types` to be explicit, add the CSS badge variant, and write unit tests. The template and filter infrastructure require zero changes ‚Äî they already loop over `grouped.keys()` and `data-ioc-type` attributes dynamically.

## Common Pitfalls

- **OTX `frozenset(IOCType)` auto-includes new enum values** ‚Äî The `frozenset(IOCType)` constructor iterates the enum at class definition time. Adding any new value to `IOCType` silently adds it to OTX's `supported_types`, and if `_OTX_TYPE_MAP` doesn't have a key for it, `lookup()` raises `KeyError`. Fix: explicit type list.

- **Email regex must not match domain-only strings** ‚Äî The `@` requirement in the email regex naturally prevents domain-only matches. But ensure the regex requires at least one character before `@` to avoid matching `@evil.com`.

- **Classifier precedence: URL before email** ‚Äî A string like `http://user@evil.com/path` should classify as URL (precedence 5), not email. The current ordering (URL at position 5, email at new position 8) handles this correctly since URL check comes first.

- **`provider_counts` loop in routes.py** ‚Äî Currently excludes CVE with `if ioc_type != IOCType.CVE`. EMAIL should also be considered for exclusion if the frontend JS expects only enrichable types in this dict. However, `"email": 0` is harmless ‚Äî the JS uses this for progress display and 0 providers means no progress expected. No change needed.
