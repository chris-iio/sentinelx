## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Phase 3 Usage |
|-------|-------|---------------|
| xs | 4px | Icon gaps in micro-bar segments, `.provider-section-header` bottom padding |
| sm | 8px | `.provider-detail-row` gap, `.no-data-summary-row` gap, `.ioc-summary-row` gap (existing) |
| md | 16px | Default element spacing (existing pattern) |
| lg | 24px | Section padding (existing `.site-main`) |
| xl | 32px | Layout gaps (existing) |
| 2xl | 48px | Major section breaks (existing) |
| 3xl | 64px | Page-level spacing (existing) |

Exceptions:
- `.verdict-label` padding target: `0.25rem 0.75rem` (4px 12px — on 4-point scale). Source: RESEARCH.md VIS-01 recommendation.
- `.verdict-micro-bar` height: `4px` — compact visual indicator on the 4-point scale (revised from 6px to align with grid).
- `.no-data-summary-row` padding: `0.5rem 0.5rem` (8px 8px — on 4-point scale). Vertical 0.5rem (8px) maintains adequate click target for a desktop-only SOC tool (revised from 0.35rem to align with grid).
- Touch target for `.no-data-summary-row` (interactive row): minimum effective click area is `width: 100%; min-height: 28px` via padding — acceptable for desktop-only SOC tool.

## Typography

Phase 3 introduces three new text roles and modifies one existing role. All other typography is inherited unchanged.

| Role | Size | Weight | Line Height | Font | Applies To |
|------|------|--------|-------------|------|-----------|
| Verdict badge (card header) | 0.875rem (≈13.1px at 15px base) | 700 | n/a (single line) | `--font-mono` | `.verdict-label` — modified from 0.7rem to create hierarchy |
| Section label | 0.65rem (≈9.75px) | 700 | n/a (single line, uppercase) | `--font-ui` | `.provider-section-header` — new element |
| No-data summary | 0.75rem (≈11.25px) | 400 | n/a (single line) | `--font-ui` | `.no-data-summary-row` — new element |

**Inherited / unchanged (not counted in Phase 3 type scale):**
- `.verdict-badge` (provider rows): 0.72rem, weight 600, `--font-mono`. This class is not modified in Phase 3. Its weight 600 is inherited from the existing design system and is not changed in Phase 3. It is excluded from the counted scale below.

**Active type scale for Phase 3 changes (4 sizes):**
1. 0.65rem — section labels (uppercase, tracked)
2. 0.72rem — provider-row verdict badges (unchanged baseline)
3. 0.75rem — no-data summary row, consensus badge (existing)
4. 0.875rem — IOC card header verdict label (upgraded from 0.7rem)

**Weights used in Phase 3 changes (2 weights):**
- 400 (regular) — no-data summary copy, micro-bar segment is chromatic (no text)
- 700 (bold) — verdict labels, section header labels

**Note on VIS-01 differentiation:** The hierarchy between `.verdict-label` (0.875rem, weight 700, `padding: 0.25rem 0.75rem`) and `.verdict-badge` (0.72rem, weight 600, `padding: 2px 8px`) is achieved by both size and padding mass. The 21% size increase plus 4x more horizontal padding creates clear visual dominance. Source: RESEARCH.md open question #3 — confirmed sufficient by research.

## Component Inventory

New CSS classes introduced in Phase 3 (all in `@layer components`):

### VIS-01: `.verdict-label` modification

No new class. Modify existing `.verdict-label` in `input.css`:
- `font-size`: `0.7rem` → `0.875rem`
- `font-weight`: `600` → `700`
- `padding`: `0.1rem 0.4rem` → `0.25rem 0.75rem`
- All other properties unchanged (border-radius, font-family, text-transform, letter-spacing, flex-shrink)

**E2E contract:** `.verdict-label` is an E2E-locked selector — it must not be renamed. Modification is safe.

### VIS-02: `.verdict-micro-bar` (new)

Replaces `.consensus-badge` production in `updateSummaryRow()`. The `.consensus-badge` CSS class and its DOM production are removed. Confirmed no E2E Playwright test queries `.consensus-badge` directly (Source: RESEARCH.md open question #2).

```
.verdict-micro-bar         — flex container, 4px height, rounded, overflow hidden, min-width 4rem
.micro-bar-segment         — height 100%, no min-width
.micro-bar-segment--malicious
.micro-bar-segment--suspicious
.micro-bar-segment--clean
.micro-bar-segment--no_data
```

`title` attribute on `.verdict-micro-bar` must encode counts in human-readable form:
`"{N} malicious, {N} suspicious, {N} clean, {N} no data"` — satisfies information density
accessibility contract (consensus readable without interaction).

### VIS-03: `.provider-section-header` (new)

Injected post-enrichment by `injectSectionHeadersAndNoDataSummary()`. Never injected per-result
(avoids `sortDetailRows()` interaction risk).

```
.provider-section-header   — 0.65rem, weight 700, uppercase, letter-spacing 0.08em
                             text: --text-muted, border-top: 1px solid --border
                             padding: 0.5rem 0.5rem 0.25rem, margin-top: 0.25rem
.provider-section-header:first-child — margin-top: 0, border-top: none
```

Text values: `"Reputation"` (verdict providers) and `"Infrastructure Context"` (context providers).
Category is derived from `CONTEXT_PROVIDERS.has(result.provider)` — no new data structure.

### GRP-02: `.provider-row--no-data` + `.no-data-summary-row` + `.no-data-expanded` (new)

```
.provider-row--no-data              — display: none (hidden by default)
.no-data-expanded .provider-row--no-data — display: flex (revealed on toggle)
.no-data-summary-row               — flex row, 0.75rem, --text-muted, cursor pointer
                                     padding: 0.5rem 0.5rem (8px vertical — on 4-point grid)
                                     border-top: 1px solid --border
.no-data-summary-row:hover         — color: --text-secondary
```

**Toggle interaction:** Click on `.no-data-summary-row` toggles `.no-data-expanded` class on
`.enrichment-details` container. `aria-expanded` attribute updated on click.

**Injection timing:** `.no-data-summary-row` is injected per-result (first no-data result creates it, subsequent ones update its count text). This approach is resilient to incomplete enrichment. Source: RESEARCH.md open question #1 — per-result injection recommended.

## Interaction Contracts

| Interaction | Trigger | Response | State |
|-------------|---------|----------|-------|
| No-data expand | Click `.no-data-summary-row` | Toggle `.no-data-expanded` on `.enrichment-details`; update `aria-expanded` | Hidden rows become `display: flex` via CSS |
| No-data collapse | Click `.no-data-summary-row` (second time) | Remove `.no-data-expanded`; rows return to `display: none` | `aria-expanded="false"` |
| Section headers appear | `markEnrichmentComplete()` fires | `injectSectionHeadersAndNoDataSummary()` scans sorted rows, inserts headers | One-time injection; headers not moved by sort |
| Micro-bar tooltip | Hover over `.verdict-micro-bar` | Native `title` tooltip shows count summary | No JS required — `title` attribute on element |

**What does NOT change in Phase 3:**
- The main IOC card expand/collapse (`.chevron-toggle` / `.enrichment-details.is-open`) — unchanged
- The `sortDetailRows()` behavior — unchanged; section headers and summary rows are not `.provider-detail-row` so they are not sorted
- All E2E-locked selectors from CSS-CONTRACTS.md — unchanged

## Copywriting Contract

| Element | Copy | Notes |
|---------|------|-------|
| Section label — reputation | `Reputation` | All caps via CSS `text-transform: uppercase`; stored as title case in DOM |
| Section label — infrastructure | `Infrastructure Context` | Matches REQUIREMENTS.md GRP-01 / Phase 4 terminology |
| No-data summary (singular) | `1 provider had no record` | Source: RESEARCH.md GRP-02 code example |
| No-data summary (plural) | `{N} providers had no record` | N ≥ 2 |
| Micro-bar tooltip | `{N} malicious, {N} suspicious, {N} clean, {N} no data` | Satisfies information density contract (consensus count readable without hover) |
| No-data section expanded aria | `aria-expanded="true"` | Set programmatically on `.no-data-summary-row` |
| No-data section collapsed aria | `aria-expanded="false"` | Default state |

**Primary CTA for this phase:** None. Phase 3 is a pure visual redesign — no user-initiated primary actions are introduced. The only new interactive element is the no-data expand toggle, which is a secondary disclosure action, not a primary CTA.

**Empty state:** Not applicable to Phase 3. IOC cards only render when IOCs are submitted; the "no IOCs" state is handled by the existing input page and is not modified in this phase.

**Error state:** Not applicable to Phase 3. Provider fetch errors display as `data-verdict="error"` rows (existing behavior). Phase 3 adds `.provider-row--no-data` class to those rows, hiding them by default like other no-data rows. The error copy ("Error") in `.provider-detail-stat` is unchanged.

**Destructive actions:** None in this phase.
