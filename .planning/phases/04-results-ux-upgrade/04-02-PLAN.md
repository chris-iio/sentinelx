<task type="auto">
  <name>Task 1: Extend VerdictEntry and refactor renderEnrichmentResult() for summary + detail rendering</name>
  <files>app/static/src/ts/modules/enrichment.ts</files>
  <action>
This is a significant refactor of enrichment.ts. The core change: instead of appending flat provider-result rows directly into .enrichment-slot, we build a summary header row (updated on each result) and route detail rows into the .enrichment-details container (sorted by severity).

**1. Extend the VerdictEntry interface (module-private):**

```typescript
interface VerdictEntry {
  provider: string;
  verdict: VerdictKey;
  summaryText: string;
  detectionCount: number;   // NEW: from result.detection_count (0 for errors)
  totalEngines: number;     // NEW: from result.total_engines (0 for errors)
  statText: string;         // NEW: the key stat string for display (e.g., "45/72 engines")
}
```

**2. Add new private helper functions:**

`computeConsensus(entries: VerdictEntry[]): { flagged: number; responded: number }`:
- Iterate entries. For each: malicious or suspicious → flagged++ and responded++. clean → responded++ only. error, no_data → skip.
- Per CONTEXT.md: no_data does NOT count as a vote. Only malicious + suspicious + clean count as "responded."

`consensusBadgeClass(flagged: number): string`:
- 0 flagged → "consensus-badge--green"
- 1-2 flagged → "consensus-badge--yellow"
- 3+ flagged → "consensus-badge--red"

`computeAttribution(entries: VerdictEntry[]): { provider: string; text: string }`:
- Filter entries to exclude no_data and error (only candidates with actual results)
- If no candidates: return { provider: "", text: "No providers flagged this IOC" }
- Sort candidates: highest totalEngines first. Ties broken by VERDICT_SEVERITY index descending (worst verdict wins).
- Return { provider: best.provider, text: best.provider + ": " + best.statText }
- "Most detailed" heuristic: VirusTotal's 72 engines always wins over providers with totalEngines=1. For providers without engine counts (GreyNoise, AbuseIPDB), totalEngines defaults to 1 from the adapter.

`getOrCreateSummaryRow(slot: HTMLElement): HTMLElement`:
- Check for existing: `slot.querySelector<HTMLElement>(".ioc-summary-row")`
- If exists, return it.
- Create div with className "ioc-summary-row".
- Insert at top of slot (before first child), AFTER the spinner-wrapper if still present. Use `slot.insertBefore(row, slot.querySelector(".chevron-toggle"))` to place it before the chevron.
- Return the new element.

`updateSummaryRow(slot: HTMLElement, iocValue: string, iocVerdicts: Record<string, VerdictEntry[]>)`:
- Get entries from iocVerdicts[iocValue]. If empty, return.
- Compute worst verdict using existing computeWorstVerdict().
- Compute attribution using computeAttribution().
- Compute consensus using computeConsensus().
- Get or create summary row via getOrCreateSummaryRow(slot).
- Clear the summary row (set textContent = "" to remove children, then rebuild).
- Build 3 child elements (all via createElement + textContent, NEVER innerHTML — SEC-08):
  a. Verdict badge: span.verdict-badge.verdict-{worstVerdict}, textContent = VERDICT_LABELS[worstVerdict]
  b. Attribution text: span.ioc-summary-attribution, textContent = attribution.text (e.g., "VirusTotal: 45/72 engines"). If attribution.text is empty (all no_data), set textContent to "No providers flagged this IOC".
  c. Consensus badge: span.consensus-badge + consensusBadgeClass(flagged), textContent = "[" + flagged + "/" + responded + "]"
- Append all 3 children to summary row.

`createDetailRow(provider: string, verdict: VerdictKey, statText: string): HTMLElement`:
- Create div.provider-detail-row, set data-verdict attribute to verdict string.
- Create span.provider-detail-name, textContent = provider.
- Create span.verdict-badge.verdict-{verdict}, textContent = VERDICT_LABELS[verdict].
- Create span.provider-detail-stat, textContent = statText.
- Append all 3 to row div, return it.

`sortDetailRows(detailsContainer: HTMLElement)`:
- Get all .provider-detail-row children as array.
- Sort descending by VERDICT_SEVERITY index of data-verdict attribute (malicious=4 first, error=0 last).
- Re-append each row to container (moves it in DOM order).
- Debounce this at 100ms per card using a module-private Map<string, ReturnType<typeof setTimeout>> keyed by ioc_value. Clear previous timer before setting new one.

`wireExpandToggles()`:
- Called once from init() (NOT per-result).
- querySelectorAll all .chevron-toggle buttons on the page.
- For each, find the sibling .enrichment-details container.
- Add click listener:
  ```typescript
  toggle.addEventListener("click", () => {
    const details = toggle.nextElementSibling as HTMLElement | null;
    if (!details || !details.classList.contains("enrichment-details")) return;
    const isOpen = details.classList.toggle("is-open");
    toggle.classList.toggle("is-open", isOpen);
    toggle.setAttribute("aria-expanded", String(isOpen));
  });
  ```
- This approach relies on .enrichment-details being the next sibling of .chevron-toggle in the template (from Plan 01).

**3. Refactor renderEnrichmentResult():**

The existing function's flow is:
1. Find card + slot
2. Remove spinner on first result
3. Track result count
4. Build provider result row
5. Route no_data to details section, others to slot directly
6. Update pending indicator, verdicts, card verdict, dashboard, sort, copy button

New flow:
1. Find card + slot (unchanged)
2. Remove spinner on first result (unchanged)
3. Add .enrichment-slot--loaded class to slot (NEW — triggers chevron visibility via CSS)
4. Track result count (unchanged)
5. Determine verdict and statText:
   - For result type: compute statText from verdict:
     - malicious: detection_count + "/" + total_engines + " engines"
     - suspicious: "Suspicious" (or detection_count + "/" + total_engines + " engines" if total_engines > 1)
     - clean: "Clean — " + total_engines + " engines"
     - no_data: "Not in database"
   - For error type: statText = result.error
6. Push to iocVerdicts with extended fields: { provider, verdict, summaryText (keep for export), detectionCount, totalEngines, statText }
7. Build detail row via createDetailRow(provider, verdict, statText)
8. Append detail row to .enrichment-details container (NOT to slot directly, NOT to nodata section). ALL results go into the details container now — no_data, errors, everything.
9. Sort detail rows in the .enrichment-details container (debounced).
10. Update summary row via updateSummaryRow().
11. Update pending indicator (unchanged).
12. Compute worst verdict, update card verdict, dashboard, sort, copy button (unchanged).

**CRITICAL CHANGES from current behavior:**
- REMOVE the no_data routing to `<details>` section. All results go into .enrichment-details container. The old `.enrichment-nodata-section` is no longer created. Delete `getOrCreateNodataSection()` and `updateNodataSummary()` functions.
- REMOVE direct slot.appendChild(providerRow) for non-no_data results. All results go into the .enrichment-details container.
- KEEP: spinner removal, pending indicator, card verdict update, dashboard update, sort, copy button, warning banner, export button, dedup.

**4. Update init():**

After the existing guard for pageResults + jobId + mode, and BEFORE starting the polling interval, call `wireExpandToggles()`. This wires all chevron buttons on the page once at init time. The handlers will work whenever the user clicks, even after results populate the details container.

**5. No changes to cards.ts — it remains unchanged.** The plan originally listed it but after analysis, no modifications are needed. The sort and dashboard functions are consumed as-is.

- Run `make js` to compile updated enrichment.ts into dist/main.js
- Run `make typecheck` to verify zero TypeScript errors
- Run `python -m pytest tests/ -v --tb=short -x` for full regression
