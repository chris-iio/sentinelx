## Interaction Contracts

Phase 1 establishes rules governing future interaction changes. No new interactions are added.

### Information Density Floor

These visible states are LOCKED — they must hold before and after every subsequent phase:

| Element | Requirement | E2E Enforcement |
|---------|-------------|-----------------|
| `.ioc-value` | Always visible in card header without any interaction | `results_page.py: ioc_values()` |
| `.verdict-label` | Always visible in card header without any interaction | `results_page.py: verdict_labels()` |
| `.ioc-type-badge` | Always visible in card header without any interaction | `results_page.py: type_badge()` |
| `.consensus-badge` | Visible without hover — not tooltip-only, not hover-only | Visual check |
| Provider detail rows | Collapsed by default; expansion is opt-in | Expected UX — not a regression |

### CSS Layer Ownership Rule

Effective from Phase 1 onward. Governs all CSS changes in Phases 2-5:

- **Component classes** (`.ioc-card`, `.verdict-label`, `.filter-btn`, `.provider-detail-row`, etc.): own ALL visual properties for existing elements. Never add Tailwind utilities alongside these on existing elements.
- **Tailwind utilities**: reserved for NEW layout structures introduced in v1.1+. Do not back-apply to elements that already have component class styles. Specificity conflict is the failure mode.

### Data-Attribute Triple-Consumer Contract

The following attributes MUST remain on the `.ioc-card` root element permanently. Moving them to child elements silently breaks three independent consumers:

| Attribute | Value Domain | Consumer 1 | Consumer 2 | Consumer 3 |
|-----------|-------------|------------|------------|------------|
| `data-ioc-value` | normalized IOC string | E2E text search filter test | `filter.ts` substring search | — |
| `data-ioc-type` | `ipv4\|ipv6\|domain\|url\|md5\|sha1\|sha256\|cve` | E2E `cards_for_type()` | `filter.ts` type pill filter | — |
| `data-verdict` | `no_data\|malicious\|suspicious\|clean\|known_good\|error` | E2E `test_cards_have_data_verdict_attribute` | `enrichment.ts: updateCardVerdict()` | `filter.ts` verdict button filter + CSS left-border color |
