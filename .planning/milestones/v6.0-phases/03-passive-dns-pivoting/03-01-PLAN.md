<feature>
  <name>ThreatMinerAdapter — passive DNS and related samples</name>
  <files>app/enrichment/adapters/threatminer.py, tests/test_threatminer.py</files>
  <behavior>
    Protocol conformance:
    - ThreatMinerAdapter satisfies Provider protocol (has name, supported_types, requires_api_key, is_configured, lookup)
    - name = "ThreatMiner"
    - requires_api_key = False
    - is_configured() returns True always
    - supported_types = frozenset({IPV4, IPV6, DOMAIN, MD5, SHA1, SHA256})

    IP lookup (rt=2 passive DNS via host.php):
    - Returns EnrichmentResult with raw_stats={"passive_dns": ["domain1.com", ...]} (list of domains that resolved to this IP)
    - Caps at 25 entries
    - Empty results or body status_code "404" -> verdict="no_data", raw_stats={}

    Domain lookup (rt=2 passive DNS + rt=4 related samples via domain.php):
    - Makes TWO sequential calls: passive DNS (rt=2) then related samples (rt=4)
    - Returns single merged EnrichmentResult with raw_stats={"passive_dns": ["1.2.3.4", ...], "samples": ["sha256hash", ...]}
    - passive_dns capped at 25, samples capped at 20
    - If first call hits 429, skip second call and return error immediately
    - Body status_code "404" on both calls -> verdict="no_data", raw_stats={}
    - Body status_code "404" on one call -> include the other call's data only

    Hash lookup (rt=4 related samples via sample.php):
    - Returns EnrichmentResult with raw_stats={"samples": ["sha256hash", ...]}
    - Caps at 20 entries
    - Results are plain string list (defensive: handle dicts by extracting values too)
    - Empty results or body status_code "404" -> verdict="no_data", raw_stats={}

    Error handling:
    - HTTP 429 -> EnrichmentError(error="HTTP 429")
    - HTTP 403 -> EnrichmentError(error="HTTP 403")
    - Timeout -> EnrichmentError(error="Timeout")
    - Unexpected exception -> EnrichmentError(error="Unexpected error")
    - Unsupported IOC type -> EnrichmentError(error="Unsupported type")
    - SSRF validation failure -> EnrichmentError with validate_endpoint message

    HTTP safety (all calls):
    - Uses requests.get with params={"q": ioc.value, "rt": rt} (NOT f-string URL)
    - timeout=TIMEOUT (SEC-04)
    - allow_redirects=False (SEC-06)
    - stream=True (SEC-05 setup)
    - read_limited(resp) for response parsing (SEC-05 byte cap)
    - validate_endpoint(url, self._allowed_hosts) before every network call (SEC-16)

    Verdict semantics:
    - All successful responses: verdict="no_data" — passive DNS is informational context, not a threat signal
    - detection_count=0, total_engines=0, scan_date=None always
  </behavior>
  <implementation>
    Follow CrtShAdapter pattern exactly. Key differences:

    1. Multi-type routing via _ENDPOINT_MAP dict mapping IOCType -> (base_url, rt_value)
    2. Shared _call() helper that does one API call with safety controls
    3. Three lookup methods: _lookup_ip(), _lookup_domain(), _lookup_hash()
    4. _lookup_domain() makes two _call() invocations (rt=2 then rt=4), merges into one result
    5. IP passive DNS results have "domain" field; domain passive DNS results have "ip" field (bidirectional)
    6. Hash related samples are plain string lists; apply defensive isinstance(r, str) check
    7. Results caps: _MAX_HOSTS=25, _MAX_SAMPLES=20

    ThreatMiner API response shape: {"status_code": "200"|"404", "status_message": "...", "results": [...]}
    CRITICAL: body status_code is a STRING "404", not int. HTTP status is always 200.

    Test pattern: patch requests.get + patch app.enrichment.adapters.threatminer.read_limited
    (same double-patch pattern as test_crtsh.py)
  </implementation>
</feature>
